# 安全策略

## 支持范围

本仓库按 `main` 分支维护，不为历史提交或个人 Fork 提供单独安全修复。

## 报告漏洞

请优先使用仓库 **Security → Report a vulnerability** 私下报告。仓库维护者需要先在 GitHub 设置中启用 Private vulnerability reporting；如果该入口不可用，请创建一个不包含漏洞细节的普通 Issue，询问可用的私下联系方式。

公开 Issue、Pull Request、讨论或日志中不要包含：

- 可复现攻击所需的敏感细节
- Token、密码、SSH 私钥、云厂商凭据或数据库连接信息
- 未公开的服务器地址、域名或运行日志

报告中建议说明影响范围、复现条件、可能的缓解方式，以及是否已经被公开利用。维护者会先确认收到报告，再协调验证、修复和披露。

## 凭据泄漏处置

如果发现真实凭据已经进入 Git 或 CI 日志，应立即在来源系统撤销或轮换凭据，并检查相关访问记录。删除当前文件或新增 `.gitignore` 规则不会让历史中的凭据失效；是否改写 Git 历史应在轮换完成后单独评估。
